# 内容编辑链路说明

本文档描述后台文章、页面在编辑过程中的自动保存、草稿、发布、版本恢复等行为，以及当前实现的设计边界。

## 一、当前做法概览

当前系统把“编辑中的未发布状态”和“已发布/可对外生效状态”拆成两层：

- `contents` 主表：保存正式内容，以及对路由/发布生效的当前状态。
- `content_revisions`：保存完整草稿快照和已发布历史版本。

核心原则如下：

- 草稿保存不应被 `slug` / `path` 的合法性或唯一性阻塞。
- 发布时才强制校验 `slug` / `path`。
- 编辑器里未发布的状态，以 `content_revisions` 里的 `draft` 记录为准。
- 对外访问、发布路由、唯一性约束，以 `contents` 主表为准。

## 二、数据职责

### `contents`

`contents` 负责：

- 文章 `slug`
- 页面 `path`
- 发布状态 `status`
- 已发布正文
- 分类、标签、SEO、评论开关等正式字段

它是“当前正式状态”的真源。

### `content_revisions`

`content_revisions` 负责：

- 当前草稿快照，`status = draft`
- 已发布历史版本，`status = published`
- 正文内容：`title`、`excerpt`、`contentType`、`contentRaw`、`contentHtml`、`contentText`
- 草稿元数据：`meta_json`

`meta_json` 当前保存的是完整草稿元数据，而不只是正文：

- 文章：`slug`、`categoryId`、`tagNames`、`coverImage`、`seoTitle`、`seoDescription`、`publishedAt`
- 页面：`path`、`template`、`seoTitle`、`seoDescription`

其中有一个重要边界：

- `allowComment`
- `showComments`
- `pinned`

这三个字段属于文章主表自身的运营属性，不进入草稿快照，也不进入历史版本。

`slug` / `path` 的规则和上面不同：

- 它们会进入草稿快照，保证刷新后仍能恢复编辑中的地址修改
- 发布时也可以进入历史版本快照，用于保留当时的发布上下文
- 但恢复历史版本时，不会把它们回滚为旧值

因此，用户即使把 `slug` 改成重复值，或把 `path` 改成非法值，也依然可以先把草稿完整保存下来。

## 三、新建内容的首次落盘

### 触发条件

新建文章/页面在编辑器中满足以下任一条件时，会触发首次自动创建：

- 标题非空
- 正文存在实际内容

对应逻辑位于：

- `src/app/(admin)/admin/_lib/draft-persistence.ts`

### 执行流程

新建文章/页面首次落盘时：

1. 调用 `/api/posts` 或 `/api/pages`，以 `createEmpty: true` 创建一条空记录。
2. 将当前完整草稿快照写入 `/api/posts/:id/draft` 或 `/api/pages/:id/draft`。
3. 成功后跳转到 `/:id` 编辑页。

### 防重复创建

编辑器内部使用 `pendingCreatedPostIdRef` / `pendingCreatedPageIdRef` 记住“已经创建但尚未跳转完成”的记录 ID。

这意味着：

- 如果首次创建后，草稿写入失败，后续自动保存不会重复 `createEmpty`
- 如果网络波动导致重试，也会继续复用同一条空记录

这样可以避免过去“反复生成孤儿空草稿”的问题。

## 四、自动保存

### 保存策略

自动保存每 5 秒检查一次，仅在以下条件同时满足时执行：

- 当前是已有 `id` 的编辑页，或新建页已触发首次创建
- 当前正文内容或元数据（`slug`/`path`、分类、标签、SEO 等）与上次自动保存时不同
- 当前没有正在进行的自动保存请求

### 保存内容

自动保存只调用 draft 接口：

- `PUT /api/posts/:id/draft`
- `PUT /api/pages/:id/draft`

保存的是完整草稿快照：

- 正文内容
- 内容类型
- 元数据

自动保存不会写 `contents` 主表，因此不会触发：

- `slug` 唯一性校验
- 页面 `path` 合法性校验
- 页面 `path` 唯一性校验

首次自动保存如果需要先创建空记录，会在后台完成以下动作：

- 创建空记录
- 保存当前完整 draft 快照
- 触发真实路由跳转，进入 `/:id` 编辑页 URL

之所以能在真实路由切换后继续保留焦点，是因为 `/new` 和 `/:id` 共用同一个 `(editor)` 路由组 layout，编辑器实例挂在持久 layout 中，不会因为 page 切换而卸载。

## 五、手动保存草稿

手动点击“保存”或“保存草稿”时，行为与自动保存保持一致：

- 已有内容：直接保存完整 draft 快照
- 新建内容：先创建空记录，再保存 draft 快照，并跳转到真实编辑页 URL

手动保存草稿同样不会写主表，因此也不会被 `slug` / `path` 校验拦住。

## 六、发布与转为草稿

### 发布

发布仍然走主表接口：

- 文章：`POST /api/posts` 或 `PUT /api/posts/:id`
- 页面：`POST /api/pages` 或 `PUT /api/pages/:id`

发布时会执行正式校验：

- 文章：标题必填，`slug` 必填且唯一
- 页面：标题必填，`path` 必填、合法且唯一

发布成功后，会额外执行两步：

1. 按本次提交内容，先写一份完整 draft 快照
2. 再把这份 draft 冻结成 `published` revision

这样做的原因是：

- 历史版本需要记录“当次真正发布出去的完整状态”
- 不能依赖用户之前某次自动保存留下的旧 draft

### 转为草稿

“转为草稿（下线）”本质上还是写主表：

- 状态切回 `draft`
- 主表进入非发布状态

它不是“保存草稿快照”的别名，而是正式状态变更。

## 七、编辑页读取规则

编辑器加载详情时，会优先读取 draft 快照。

具体表现为：

- 若存在 `draftContent`，则标题、正文、`slug/path`、SEO、模板等内容字段以 draft 为准
- 若不存在 draft，则回退到主表字段

文章的评论开关与置顶状态始终读取主表，不读取 draft。

这样可以保证：

- 刷新页面不会丢失未发布修改
- 非法或重复的 `slug/path` 也能在编辑器里稳定恢复

## 八、后台列表读取规则

后台列表不是完全基于 draft 渲染，而是做了有限叠加：

- 文章列表会优先显示 draft 标题
- 页面列表会优先显示 draft 标题、`path`、`template`、`contentType`

这样可以让刚保存的草稿在列表中看起来更接近用户的当前编辑状态。

后台搜索会同时匹配主表标题和草稿标题，因此用户修改标题后即使未发布，也能在列表搜索中命中。

### 排序规则

- 文章列表：置顶 → 从未发布过的草稿置顶 → 发布时间倒序 → 创建时间倒序
- 页面列表：从未发布过的草稿置顶 → 发布时间倒序 → 创建时间倒序

"从未发布过"指 `publishedAt` 为空。已发布后又转为草稿的内容仍保留 `publishedAt`，按正常时间排序。

## 九、历史版本与恢复

### 历史版本来源

历史版本来自“发布时冻结的 draft 快照”，因此它包含：

- 当时的正文
- 当时的元数据

### 恢复版本

恢复版本时，流程是：

1. 找到指定的 `published` revision
2. 用它创建一份新的 draft
3. 立即将该 draft 冻结为新的 `published` revision
4. 同步更新主表

同步主表时：

- 文章会恢复分类、标签、头图、SEO、正文
- 页面会恢复 `template`、SEO、正文

评论开关和置顶状态不会随版本恢复而回滚，因为它们不属于版本化内容。
`slug` / `path` 也不会随版本恢复而回滚，因为当前线上地址可能已经被引用。

这保证“恢复版本”恢复的是内容状态，而不是运营状态。

## 十、维护要点

后续修改这条链路时，建议始终遵守以下规则：

### 1. 草稿接口不要做发布级校验

`/draft` 接口的职责是保存编辑态，不是判定内容是否可发布。

不要在 draft 接口里加入：

- 文章 `slug` 唯一性校验
- 页面 `path` 合法性/唯一性校验

否则会重新把“正文保存能力”绑死在路由字段上。

### 2. 发布接口必须保持严格

主表接口仍然要严格校验发布条件，否则会把非法状态带入正式路由层。

### 3. 新建态必须复用待创建 ID

如果修改了新建流程，必须保留“失败重试复用同一条空记录”的机制，否则容易再次出现批量孤儿草稿。

### 4. draft 接口必须校验 content 存在性

`/draft` 接口在写入 revision 前会校验对应的 content 是否存在（已删除的也视为不存在），不存在返回 404。这可以防止为不存在的 content 写入孤儿 revision。

### 5. 发布时历史版本必须基于本次提交生成

不要直接冻结旧 draft。发布时应先把本次提交写成最新快照，再冻结成历史版本。

### 6. 新增可编辑元数据时，需要同时改四处

如果后续新增了可编辑字段，例如文章副标题、页面布局配置等，需要先判断它属于“内容属性”还是“运营属性”。

只有内容属性才应该进入 revision 草稿和历史版本。确认属于内容属性后，再同步修改：

1. draft metadata 定义
2. draft 保存 payload
3. draft 读取回填逻辑
4. 版本恢复同步逻辑

否则会出现“能编辑但保存后丢失”或“恢复版本不完整”的问题。

## 十一、已知设计边界

### 1. 单槽位 draft

当前 draft 为单槽位设计：每个内容仅保留一份 draft 记录（通过 upsert 实现）。如果用户 A 和用户 B 同时编辑同一篇文章，后保存的 draft 会覆盖先保存的。当前不支持多用户并发编辑同一篇内容。

### 2. `createdBy` 与 `updatedBy`

`content_revisions` 表中，`createdBy` 仅在首次创建 draft 记录时写入，表示草稿的初始创建者。后续自动保存和手动保存只更新 `updatedBy`，表示最后一次操作的用户。发布冻结（`publishDraft`）时也会更新 `updatedBy`，记录执行发布操作的用户。因此 `updatedBy` 在已发布版本中代表的是"谁执行了发布"，而不仅仅是"谁最后编辑了草稿"。

### 3. 删除操作的事务保证

`deletePost()` 和 `deletePage()` 将软删除（设置 `deletedAt`）和关联 revision 清理（`deleteRevisionsByTarget`）包裹在同一事务中，保证两步操作要么同时成功，要么同时回滚。

### 4. 孤儿 revision 清理

尽管 draft 接口做了存在性校验，在极端竞态下（自动保存请求已通过校验，content 随后被另一个请求删除）仍可能产生孤儿 revision。系统通过每日定时任务（`src/cron/1d.ts`）调用 `cleanOrphanRevisions()` 清理所有无对应 content 记录的 revision。软删除的 content 对应的 revision 不会被清理。

## 十二、推荐理解方式

可以把当前模型理解为：

- `contents`：线上正式状态
- `content_revisions.status = draft`：编辑器当前工作区
- `content_revisions.status = published`：发布历史快照

这个分层比“草稿直接写主表”更适合继续演化，因为它把编辑态和正式态分开了，也为后续做更完整的版本能力留出了空间。
